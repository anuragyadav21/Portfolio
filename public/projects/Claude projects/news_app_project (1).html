This project uses a validation-first pipeline: ingest live NYT feeds, summarise by section, run a 3-agent workflow that cross-links stories, scores world mood, and cross-references the narrative against live Yahoo Finance data before anything is shown to the user.

System Architecture

The app is built on Python Shiny (ASGI-compatible, deployed via uvicorn). The architecture separates concerns cleanly: data ingestion, article enrichment, agentic orchestration, and research are distinct modules with defined interfaces.

02

Data Flow

Articles enter through parallel NYT section fetches, are enriched per-article, then feed both the card UI and the agentic pipeline. The flow below traces every data path from API key configuration to rendered output.

1 · Ingest and enrich

NYT sections are fetched in parallel, classified, then each article gets sentiment, impact, and tone-configurable summaries.

flowchart TD
    NYT[NYT Top Stories API] --> FETCH[data_fetch.py]
    KEYS[API keys in .env] --> FETCH
    FETCH --> CAT[categorization.py]
    CAT --> AI[ai_services.py]
    CAT --> IMP[impact_classifier.py]
    OAI[OpenAI API] --> AI
    OAI --> IMP
    AI --> ENR[Enriched article records]
    IMP --> ENR

2 · UI and on-demand research

Enriched records drive the card grid; “Dive deeper” triggers a separate tool-calling loop (Wikipedia + Yahoo Finance).

flowchart LR
    ENR[Enriched articles] --> CARDS[News cards UI]
    ENR --> BRIEF[Section briefs]
    CARDS --> DIVE[Dive deeper]
    DIVE --> RES[research_agent]
    OAI2[OpenAI API] --> RES

3 · Signal Studio pipeline

Section briefs feed a sequential 3-agent workflow; Agent 3 cross-checks narrative against a live yfinance snapshot (QC contribution).

flowchart LR
    BRIEF2[Section briefs] --> A1[Agent 1 cross_section]
    A1 --> A2[Agent 2 world_sentiment]
    A2 --> A3[Agent 3 market_validation]
    MKT[market_data.py] --> A3
    A3 --> SIG[Signal Studio]
No vector database: "RAG" here means retrieval at request time from Wikipedia and Yahoo Finance endpoints, plus structured market history from yfinance in market_data.py. There is no separate vector store. This is a deliberate architecture choice: freshness over recall, real-time grounding over indexed similarity.
03

Three-Agent Pipeline

The pipeline runs sequentially. Each agent consumes the previous agent's output plus the original section packets. The workflow is orchestrated by agents/workflow.py, which first generates section briefs then fires the three-agent chain.

Agent 1
cross_section_agent
Input: section packets (headlines, summaries, sentiment counts per tab). Finds thematic connections across sections, a business story that connects to a world development, a technology story that links to policy. Output: cross-section link map passed to Agent 2.
→
Agent 2
world_sentiment_agent
Input: Agent 1 output + original section packets. Produces world mood score and editorial stance, is the aggregate news narrative optimistic, neutral, or negative? What is the dominant theme of the day's coverage? Output fed to Agent 3.
→
Agent 3 ★
market_validation_agent
Input: Agent 2 output. Compares news narrative to live market data via tool calling: get_market_snapshot(symbols) → fetches S&P 500, NASDAQ, Dow, Gold, Oil, BTC. Flags divergence: if narrative says "concern" but markets are up, that contradiction is surfaced explicitly. Fallback: JSON mode with prefetched snapshot if tool path fails.
Market universe6^GSPC · ^IXIC · ^DJI · GC=F · CL=F · BTC-USD
Market cache TTL10 minWhen no symbol list passed
Tool fallbackJSON modePrefetched snapshot when tool path fails
Pipeline patternSequential1 → 2 → 3 · shared context
Why sequential and not parallel? Agent 2 needs Agent 1's cross-section links to produce an accurate mood score. Agent 3 needs Agent 2's narrative framing to know what to validate against the market data. The dependencies are real, parallelising would require each agent to independently reconstruct context that the previous agent already computed.

Tool Calling Implementation

Two distinct tool-calling contexts: the research agent (per-article deep dives) and the Signal Studio market validation tool. Both use OpenAI's function calling format. Tools defined in the API call, tool_calls parsed from responses, results appended as role:tool messages.

Research agent loop (research_agent/agent.py, run_research_brief):

Agent loop, max 6 rounds 
# Post to Chat Completions with tools defined
response = openai.chat.completions.create(
    model="gpt-4o",
    messages=messages,
    tools=[wikipedia_lookup_schema, yahoo_finance_quote_schema]
)

# Handle tool_calls in response
while response.choices[0].finish_reason == "tool_calls":
    tool_calls = response.choices[0].message.tool_calls
    for tc in tool_calls:
        result = dispatch_tool(tc.function.name, tc.function.arguments)
        messages.append({
            "role": "tool",
            "tool_call_id": tc.id,
            "content": result
        })
    response = openai.chat.completions.create(...)  # next round
    rounds += 1
    if rounds >= MAX_ROUNDS: break

05

UI Architecture: Quick-then-Full Pattern

The app uses a deliberate two-pass rendering strategy that treats LLM latency as a first-class design constraint. The same way infrastructure systems treat uncertainty as a design input rather than an implementation afterthought.

Pass 1: Immediate render (deterministic)
On first paint: deterministic placeholders for all card positions, fast workflow snapshot with structural metadata (section counts, article IDs, timestamps). User sees content immediately. No blank-screen wait. The UI is interactive before any LLM call completes.
Pass 2: Background enrichment (LLM-driven)
Background process replaces placeholders with: full AI-generated section briefs per tab, complete 3-agent Signal Studio result, sentiment scores and impact labels per article. Each replacement is a reactive update. The Shiny reactive graph handles ordering automatically.
Signal Studio rendering
ui/agent_views.py renders the workflow JSON from agent_workflow_state — the reactive variable in app.py that the workflow populates. Agent outputs are structured JSON: Agent 1 links, Agent 2 mood score, Agent 3 market comparison. All rendered as formatted sections in the Signal Studio tab.
Research modal (Dive deeper)
Each article card exposes a "Dive deeper" trigger that fires run_research_brief(article_url, article_context). The research agent result is rendered in a modal overlay, cached per URL so repeated clicks on the same article are instant.

08

Reflection

Most of the engineering work here is outside the model calls: reliable NYT ingestion, caching, fallbacks when tool calling fails, and checking the agent narrative against Yahoo Finance before anything reaches the UI.

V&V applied to an AI product
Agent 3 compares the generated narrative to live market data before display. That is the same cross-check pattern used in the Gemma IV&V project, applied here as a small production step rather than a separate test harness.
What building production AI taught me
Failures showed up at interfaces: cache keys, tool-call parsing, and JSON fallback when function calling failed. Latency shaped the quick-then-full UI pattern; without it, users waited on a blank screen while the three-agent chain finished.
Why the QC layer is rare
Many demos stop at model output. Agent 3 adds a short market-validation pass (a few dozen lines of Python) so obvious narrative–market mismatches are caught before render.
