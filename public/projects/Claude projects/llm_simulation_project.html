01

Introduction

Large Language Model inference systems process user queries through two distinct computational phases. Prefill handles the input prompt, highly parallelizable across tokens. Decode generates output tokens sequentially, memory-bandwidth constrained and difficult to batch across unrelated queries.

This asymmetry makes GPU scheduling non-trivial. A scheduler that optimizes for simplicity may leave expensive parallel capacity idle. A scheduler that batches aggressively may improve throughput while degrading tail latency for individual users.

Two-phase inference pipeline

Systems framing: The project treats LLM serving as a queueing system with heterogeneous service times across phases, not as a single "inference call" with one latency number.

Scheduling Strategies

Basic Scheduler

Assigns one query at a time to the next idle GPU worker. Simple FIFO queue discipline. Each query runs full prefill then full decode before the worker accepts another assignment.

BASELINE · LOW COMPLEXITY

Chunked Scheduler

Batches multiple queries and processes prefill in configurable chunks. Designed to keep GPU utilization high during the parallelizable prefill phase. Chunk size is a tunable parameter.

05

Performance Metrics

Time-to-First-Token (TTFT)
Elapsed time from query arrival until the first output token is generated. Primary measure of user-perceived responsiveness. Sensitive to queue depth, prefill batching, and worker availability.
Time-Between-Tokens (TBT)
Average interval between consecutive decode tokens. Captures streaming quality during generation. Dominated by decode-phase service time once prefill completes.
Arrival & finish times
Raw event timestamps logged per query enable reconstruction of queue wait time, service time, and total system time, supporting deeper post-hoc analysis beyond summary statistics.
Replication index
Each trial tagged by replication number (0 to replications−1). Enables distribution-level comparison across stochastic arrival sequences rather than point estimates.

08

Tradeoffs & Limitations

Scheduler trade space

Utilization vs latency
Chunked batching improves GPU utilization during prefill but may increase queue wait time for individual queries under high arrival rates, visible in TTFT tails.
Model abstraction
The simulation uses parametric cost curves, not real GPU kernel measurements. Results show scheduling dynamics, not absolute production latency predictions.
Homogeneous workers
All GPU workers share the same cost parameters. Real clusters have heterogeneous hardware, memory limits, and model sharding, not captured here.
No network / KV-cache effects
Prefill-decode pipeline is modeled without KV-cache reuse, prefix caching, or cross-node communication, relevant in production vLLM/TGI deployments.
09

Conclusions

Measured at 10 GPUs, Chunked scheduling cuts P99 TTFT by ~48% (524.7 → 271.5 ms) while leaving medians unchanged. The operational lesson is that batching buys tail latency under contention, not average case at moderate load.

Saturation sweeps show Chunked sustaining higher offered load before collapse (peak throughput ~6.06 q/s vs ~1.79 q/s on Basic with 1 GPU). Worker scaling confirms chunked wins are largest at 1 GPU (P99 3579 vs 1112 ms). The DES + CSV pipeline (warmup stripping, replication aggregation) is the reusable artifact for comparing schedulers before production deployment.
