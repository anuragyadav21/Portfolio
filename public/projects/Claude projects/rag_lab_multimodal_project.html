04 · FAQ

Common questions

Which API?

The project is configured for OpenAI as the primary provider (OPENAI_API_KEY, default models gpt-4o-mini for text and gpt-4o for vision). Groq remains an optional fallback when OPENAI_REQUIRED is disabled. Retrieval uses Chroma default embeddings locally, not an external embedding API.

Limitations

Native PowerPoint shapes (not pasted images) may not export to slide PNGs unless you use PDF export or LibreOffice headless conversion. Equation detection uses heuristics (LaTeX markers, not full OCR of math). Quality depends on re-ingesting after pipeline changes (ingest.py --reset).

How to run

Install dependencies, copy .env.example to .env with API keys, place files under the documents folder, run ingest, then streamlit run app.py or python query.py with a question. Optional: visualize_embeddings.py for a 2D PCA plot of the embedding space.

01

Problem & systems framing

Systems engineering courses deliver knowledge as structured narratives: definitions on early slides, trade-space plots mid-lecture, and exam-style synthesis at the end. Students need fast, trustworthy access to what the slides actually say and what figures depict—not generic web answers.

RAG Lab treats the lecture corpus as the single source of truth. The architecture separates ingestion (deterministic extraction and indexing), retrieval (similarity search plus optional graph expansion), and generation (LLM synthesis with citations). That split keeps the system auditable: every answer can be traced to retrieved chunks and slide references.

Design principle: Ground first, generate second. The LLM is instructed to answer only from supplied context and to state when evidence is missing, reducing confident hallucination on course-specific questions.

Query orchestration & modular workflow

When a user submits a question, the system runs a fixed orchestration script with no agent planner. Each box below is a Python module role with a clear input/output contract; only the final step calls an external LLM API.

Offline vs online paths

Offline (once): ingest.py reads course files, extracts text and slide images, embeds chunks into Chroma, then rebuilds memory_graph.json. Online (every question): only the query is embedded for search; stored chunk vectors are not recomputed. The graph file is read to add up to two related chunks, then context goes to OpenAI.

Vector search detail

Chroma embeds the query with the same default function used at ingest, then returns the nearest chunk documents and metadata by cosine distance.

sequenceDiagram

    participant EF as "Embedding function"
    participant CH as Chroma collection

    Q->>EF: encode query
    EF->>CH: query with n_results k
    CH->>CH: HNSW cosine search

Honest scope: RAG Lab does not run a multi-LLM agent framework (no planner, tool loop, or inter-agent messaging). Orchestration is explicit function calls in app.py → retrieve_hits → synthesize_multimodal_answer.
05

Retrieval & memory graph

Queries hit Chroma with cosine space on the default embedding function. retrieve_hits normalizes metadata: resolves image_path to an on-disk file, coerces slide numbers, and extracts inline LaTeX spans for the UI.

The memory graph stores concept → chunk keys and co-occurrence edge weights. After top-k vector hits, up to two neighbor chunks sharing concepts with the query hits may be appended, conservative expansion to limit context drift.

References in the UI: Answers show compact reference links; clicking opens a popover with full chunk text, optional slide image, equation rendering, and an explain-diagram action when an image is present.
06

LLM & multimodal generation

Configuration loads from .env via python-dotenv. When OPENAI_API_KEY is set (and OPENAI_REQUIRED=1 by default), all completions use the OpenAI client. Text answers use structured system prompts (direct answer, concepts, equations, diagram interpretation, takeaways). Vision paths attach base64 data URLs for slide PNGs.

If the vision model call fails with a recoverable error, the system can fall back to text-only completion using slide text, preserving a response when images cannot be processed.

Limitations & caveats

Slide image coverage
PPTX without embedded pictures may lack PNG previews unless exported to PDF with PyMuPDF or converted via LibreOffice (soffice) for full-slide renders.
Equation handling
has_equation and LaTeX spans rely on heuristics ($...$, \frac, etc.). Slides with rasterized math may not render as typeset equations.
Embedding model
Chroma default embeddings are local and general-purpose; domain-specific retrieval quality may improve with a dedicated embedding model (not implemented in this repo).
Index freshness
After changing extraction or graph logic, run ingest.py --reset so metadata, slide images, and the memory graph stay aligned.
09

Conclusions

RAG Lab is a course-grounded multimodal assistant: lecture files become searchable vector chunks with slide metadata, retrieval combines embedding similarity with a lightweight concept graph, and OpenAI text/vision models synthesize answers tied to explicit references.

The system is designed for auditability (popover references with full chunks), multimodal study (diagram and equation tabs, explain-diagram actions), and repeatable indexing (hash-based incremental ingest with graph rebuild).

For SYSEN 5400-style workflows, it supports both exploratory chat and structured sidebar outputs, notes, summaries, flashcards, and practice questions, without leaving the indexed corpus.

Operational requirements are modest: Python dependencies in requirements.txt, local Chroma storage, and API keys in .env. No separate embedding API is required for the default path.

Future improvements could include stronger slide rendering for native shapes, dedicated math OCR, and embedding model choice tuned to engineering notation, without changing the core ingest → retrieve → generate structure.

Bottom line: A local, multimodal RAG stack over ~3k slide-aware chunks and 77 source files, OpenAI for grounded answers, Chroma for retrieval, and a Streamlit front end for study workflows.
